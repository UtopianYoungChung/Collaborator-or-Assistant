3.3.2 Phase Definitions

Phases are assigned using a temporal state machine implemented in src.analysis.workflows.assign_phases_temporal. Terminal outcomes are distinguished as Merged and closed vs Unmerged and closed (event type merged vs closed).

Phase / Terminal Entry condition Exit condition Events captured
Initiation PR created First committed or closed assigned, labeled, early review_requested
Development First committed event in PR timeline First review_requested after commits, or merged/closed committed, commented, head_ref_force_pushed
Review review_requested reviewed:CHANGES_REQUESTED (→ REVISION) or merged/closed reviewed, commented, review_requested
Revision reviewed:CHANGES_REQUESTED Next review_requested (→ REVIEW) or merged/closed committed, commented
Merged and closed merged — (terminal) —
Unmerged and closed closed — (terminal) —

Initiation may be empty in observed event order (e.g. first event is already committed). Edge case: PR closed before any commit → Initiation → Unmerged and closed. Development entry: the first committed is the first commit event in the PR event stream; it is usually part of the source branch when the PR was opened, not necessarily a commit made after the PR was created. Review: APPROVED does not end the PR; workflow remains in Review until a terminal event. A PR without a formal review_requested event goes from Development directly to merged/closed. Revision: a revision cycle is counted only when CHANGES_REQUESTED is followed by ≥1 committed before the next review_requested. Terminal states are absorbing (Merged and closed = PR was merged; Unmerged and closed = PR was closed without merge).
